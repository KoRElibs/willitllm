# meta/benchmarks

Empirical measurements from real hardware used to calibrate the speed estimation formula in
`app.calc.js`. Files are generated by `meta/scripts/benchmark.py` — do not create them manually.

---

## Running a benchmark

```bash
python3 meta/scripts/benchmark.py \
  --gpu "RTX 3090" \
  --vram 24 --bandwidth 936 \
  --model devstral:24b \
  --kv-type f16 \
  --fill --runs 2
```

`--gpu` is required to save a file. Without it, results print to console only.
`--fill` pads the prompt to the full context size — slow but realistic (matches Cline usage).
Without `--fill`, the prompt is ~50 tokens regardless of `num_ctx`; fast but not realistic.

Default context sweep (when `--ctx` not specified): `default · 4k · 8k · 32k · max`
where `max` is the willitllm formula result for the given VRAM/model/KV combination.

The script auto-detects model architecture and weights size from `/api/show`. Pass
`--weights-gb` to override if detection fails.

---

## Naming convention

Files are named automatically by the script:

```text
{gpu-slug}-{model-slug}-{kv-type}.md
```

- `gpu-slug` — `--gpu` value lowercased, non-alphanumeric → hyphens (`RTX 3090` → `rtx-3090`)
- `model-slug` — `--model` value lowercased, same substitution (`devstral:24b` → `devstral-24b`)
- `kv-type` — `--kv-type` value: `f16`, `q8_0`, or `q4_0`

## File structure

```text
# {GPU} · {model} · {kv-type}

Date: YYYY-MM-DD  |  Fill: yes/no  |  Runs: N
```

**Fill:** `yes` = prompt padded to full context (realistic); `no` = short prompt only (fast).
**Runs:** number of benchmark runs averaged per context size.

### Setup table

Documents the exact configuration. `Max ctx` is computed by the script using `OVERHEAD_GB = 0.5`
— note that `app.calc.js` uses `OVERHEAD_GB = 0.8`, so the script's predicted max is optimistic
relative to what the website shows users.

### Measured table

| Column | Meaning |
| --- | --- |
| Ctx | Context label (`4k`, `32k`, etc.) |
| Prompt tok | Actual input tokens processed (from ollama `prompt_eval_count`) |
| KV GB | KV cache VRAM at this context (`prompt_tok × bytes_per_token`) |
| Gen avg | Average generation speed in t/s across runs |
| Prefill | Prompt processing speed in t/s |
| VRAM | Total VRAM in use as reported by `/api/ps` |

`⚠ SPILL` means some model layers were offloaded to system RAM — generation collapsed.
This marks a hard OOM boundary and is the primary data point for calibrating `OVERHEAD_GB`.

### vs willitllm table

Compares measured generation speed against two formula variants at each context size:

| Column | Meaning |
| --- | --- |
| Measured | Actual `Gen avg` t/s |
| Old est | `bandwidth × gen_eff / weights_gb` — formula that ignores KV cache in denominator |
| New est | `bandwidth × gen_eff / (weights_gb + kv_gb)` — formula that includes KV in denominator |
| Old err | How wrong the old formula was (`old_lo / measured − 1`, negative = under-predicted) |
| New err | Same for new formula |

`gen_eff [lo, hi]` shown above the table is the current range from `QUANT_INFO` for that
quantization. These are the efficiency bounds being validated.

The script also prints a "willitllm hint" to the console (not saved to the file): observed
`gen_lo` and `gen_hi` across all runs, formatted as `data.quantizations.js` entries.

## What these benchmarks calibrate

| Constant | Location | How benchmarks inform it |
| --- | --- | --- |
| `OVERHEAD_GB` | `app.calc.js` | SPILL rows reveal the true usable VRAM boundary. Was raised 0.5 → 0.8 after `gtx-1660-super-llama3-2-3b-f16.md` showed spill at a context the 0.5 GB formula predicted would fit. |
| `DECODE_ATTN_EFF` | `app.calc.js` | Controls how fast generation speed falls with context. Calibrated from RTX 3090 f16/q8_0/q4_0 sweeps — `SPEC.md §5.2` describes the model. |
| `gen_eff [lo, hi]` | `data.quantizations.js` per quant | Compare `Gen avg` against `New est` range. If measurements fall outside the range, update the bounds. |
| `prefill_eff [lo, hi]` | `data.quantizations.js` per quant | `Prefill` column — prompt processing speed bounds. |

Before changing any of these constants, read the benchmark files that originally motivated the
current value. Update `SPEC.md §4–5` and `meta/BUGS.md` after any change.

---

## Known inconsistency

The script uses `OVERHEAD_GB = 0.5` for its `Max ctx` calculation; `app.calc.js` uses `0.8`.
The `Max ctx` line in saved benchmark files is therefore more optimistic than what the website
shows. Keep this in mind when comparing the script's predicted max against user-facing results.
